## 源码研读 Round 2 - openclaw v2026.4.2

### 核心架构
**Gateway 中心化 + 插件化**架构：
- 单长生命周期 Gateway 进程（WS 服务，默认 127.0.0.1:18789），统一管理所有聊天通道
- 通道协议（WhatsApp/Telegram/Discord 等）全部插件化，通过 `plugin-sdk` 暴露 typed hooks
- 通道 → Binding 路由层（消息按 channel/accountId/peer 匹配到对应 agent）
- Agent Runtime 是 Pi agent core 的 wrapper，session 管理/QMD memory/bindings 是 OpenClaw 自有层

### 关键文件
- `docs/concepts/architecture.md`：Gateway 架构总览，WS 协议（req/res/event 三帧），节点/客户端连接模型
- `docs/concepts/multi-agent.md`：多 agent 隔离模型（workspace + agentDir + sessions + auth-profiles 各自独立）
- `docs/concepts/agent.md`：Bootstrap 注入（SOUL.md/AGENTS.md 等文件直接注入 context）+ Skills 三层加载
- `plugin-sdk/`（type definitions）：80+ 导出子模块，channel-runtime / approval-runtime / reply-runtime 等
- `dist/`：编译产物，bundle 架构（channel/plugins/infra 分层）

### 可复用点
1. **Bootstrap 文件注入**：workspace 里的 SOUL.md/AGENTS.md/USER.md 直接注入 context → 我已借鉴（TOOLS.md）
2. **TypeBox 协议驱动**：schema → JSON Schema → codegen → Swift 模型，协议即文档
3. **Binding 路由**：most-specific-wins 语义 + 优先级 tier 设计，比 if-else 清晰得多
4. **Skill 三层加载**：bundled / `~/.openclaw/skills` / workspace/skills，workspace 优先冲突解决
5. **Self-contained 单文件 CLI**：openclaw.mjs 入口，极简 bootstrap

### 差距分析
OpenClaw 是 thousands-of-lines 工业级 Gateway，scope 差异大。我的借鉴价值：
- Pi agent core 是黑盒（npm package 无源码），无法深入理解
- Plugin SDK 是完整 type-safe 扩展系统，我的 skills 缺乏 typed schema/contract
- Binding 路由的 tier 优先级设计可以用到我的内部路由逻辑
- TypeBox schema → JSON Schema → codegen 的思路可用于技能定义规范化
